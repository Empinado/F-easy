# 🎓 Итоговый проект — Анонс

> Итоговый проект — это ваша возможность показать всё, чему вы научились: работу с фронтендом, бэкендом и базой данных в рамках одного реального продукта. Именно этот проект вы будете защищать перед экспертами из ведущих компаний.

> 🏆 **Итоговый проект весит значительно больше, чем домашние задания.** Домашние задания — это тренировка и закрепление материала лекций. Главное в ДЗ — применять знания из лекций на практике. Проект — это демонстрация того, что вы умеете работать в комплексе: проектировать, писать код, строить архитектуру и представлять результат. Именно проект формирует основную часть итоговой оценки.

## Что нужно сделать

Вы разрабатываете полноценное веб-приложение **на тему по вашему выбору** — это может быть магазин, CRM, платформа, онлайн-игра, сервис бронирования — что угодно, что решает реальную задачу.

На реализацию у вас **10 недель**. Объём проекта рассчитывайте исходя из этого срока.

**Даты защиты:** 4 июля или 11 июля.

## Технические требования

### Обязательный стек

| Слой | Технология |
|------|-----------|
| Фронтенд | React + TypeScript |
| Бэкенд | Node.js (Python — по согласованию, см. FAQ) |
| База данных | PostgreSQL |

### Функциональные требования

#### 🟢 Минимум — без этого проект не принят

- Аутентификация и авторизация пользователей
- Личный кабинет: редактирование данных, смена пароля, история действий пользователя
- CRUD-операции над основными сущностями
- Поиск и фильтрация данных
- Пагинация на списках
- Обработка ошибок и валидация на фронте и бэкенде

#### 🟡 Хорошо — ожидаемый уровень

- Роли пользователей (минимум: user / admin) и admin-панель для управления контентом и пользователями
- Интеграция с внешним API (Dadata, 2GIS / Google / Yandex Maps и т.д.)
- Понятный и аккуратный UI
- Чёткое разделение ответственности в коде (слои, модули)
- Осмысленная схема базы данных

#### 🔵 Отлично — выделит проект на защите

- Статусная модель сущностей (например: заказ → создан → оплачен → доставлен)
- Email или in-app уведомления (подтверждение регистрации, изменение статуса и т.д.)
- Unit и/или интеграционные тесты (фронт и/или бэк)
- Нетривиальная бизнес-логика
- Продуманный UX, адаптивность

## Технологические требования

Это требования к **культуре разработки** — они не описывают бизнес-логику, но влияют на оценку инженерной работы.

#### 🟢 Минимум

- Проект хранится в Git-репозитории
- Конфигурация и секреты вынесены в `.env`, не хардкодятся в коде
- Есть `README.md` с описанием проекта и инструкцией по локальному запуску

#### 🟡 Хорошо

- Осмысленная Git-история: коммиты по фичам, не один коммит на весь проект
- Ветки под отдельные задачи / фичи. Вливайте через Pull-Request. Делайте сами себе ревью

#### 🔵 Отлично

- Документация API (Swagger / OpenAPI)
- Деплой приложения (хотя бы dev-окружение через Docker или облачный хостинг)
- Архитектурные схемы и схемы бизнес-процессов (схема архитектуры системы, диаграммы последовательностей, схемы бизнес-процессов в нотации BPMN или иной). Наличие таких схем — огромный плюс: хорошо оформленная визуальная документация может заменить большой объём текстового описания и произвести сильное впечатление на защите. В пределе документация может состоять только из схем и живой демонстрации проекта.

## Как выбрать идею

Это один из самых сложных шагов. Вот несколько способов, которые помогут.

### 💡 Откуда брать вдохновение

**Из личного опыта:**
- Что раздражает вас в существующих сервисах?
- Какой задачи вам не хватает в повседневной жизни?
- Что вы делаете вручную, хотя можно автоматизировать?

**Из популярных категорий:**

| Категория | Примеры идей |
|-----------|-------------|
| 🛒 Маркетплейс | Магазин одежды, техники, handmade-товаров |
| 📅 Бронирование | Запись к врачу, аренда, запись в барбершоп |
| 🎮 Онлайн-игры | Многопользовательская стратегия, карточная игра с лобби и матчмейкингом, браузерный рогалик с таблицей рекордов, викторина в реальном времени против других игроков |
| 📚 Обучение | Трекер привычек, платформа с курсами, карточки слов |
| 💼 CRM / управление | Таск-менеджер, CRM для небольшого бизнеса |
| 🏠 Объявления | Аренда жилья, доска объявлений, найти соседа |
| 🍕 Еда | Доставка, планировщик питания, рецепты с фильтрами |
| 💬 Мессенджер / чат | Групповые чаты с каналами (как Slack/Discord), личные переписки, чат поддержки с ботом, анонимный чат по интересам |
| 📹 Видеосервис | Платформа для загрузки и просмотра видео с комментариями и подписками, стриминговый сервис с каталогом и историей просмотров |
| 📞 Коллаборация | Онлайн-доска для совместной работы, сервис для созвонов с записью встреч и расшаровкой экрана, совместное редактирование документов |
| 🏥 Здоровье | Дневник симптомов, трекер питания и активности, платформа для записи к специалистам |
| 🌍 Сообщество | Форум, сервис отзывов, Q&A-платформа, платформа волонтёров |

### 🛠 Инструменты для формирования идеи

**Для генерации и мозгового штурма:**
- [ChatGPT](https://chat.openai.com) / [Claude](https://claude.ai) — опишите свои интересы, попросите 10 идей для проекта с описанием сущностей
- [Miro](https://miro.com) — mind map для связей между сущностями и фичами
- [FigJam](https://www.figma.com/figjam/) — доска для брейншторминга

**Для исследования рынка:**
- [Product Hunt](https://www.producthunt.com) — посмотрите, что уже делают стартапы
- [VC.ru](https://vc.ru) — статьи о продуктах и бизнес-задачах
- [Google Trends](https://trends.google.com) — актуальность темы

**Для прототипирования и дизайна:**
- [Figma](https://figma.com) — нарисуйте 2–3 экрана до начала кода
- [Excalidraw](https://excalidraw.com) — быстрые схемы и wireframes
- [Mobbin](https://mobbin.com) — готовые референсы UI для вдохновения

**Для планирования задач:**
- [Notion](https://notion.so) — задачи + документация в одном месте
- [Trello](https://trello.com) — простая Kanban-доска
- [Linear](https://linear.app) — продвинутый трекер задач, как в реальных командах

### ✅ Чеклист готовности идеи

Прежде чем приступать к разработке, ответьте на эти вопросы:

- [ ] Я могу объяснить идею одним предложением
- [ ] Я понимаю, кто пользователь и какую задачу он решает
- [ ] Я перечислил основные сущности (User, Product, Order и т.д.)
- [ ] Я понимаю, какие CRUD-операции нужны
- [ ] Я оценил, что успею за 10 недель
- [ ] У меня есть задачи в трекере (хотя бы крупными блоками)

## Контрольные точки

Каждые **1.5–2 недели** вы списываетесь/созваниваетесь(Если сможете их вытащить на созвон) с ментором. Контрольные точки разделены на две фазы: сначала **проектирование** (думаем и рисуем), потом **реализация** (пишем код).

> ⚠️ Не начинайте писать код до завершения точки 2. Плохо спроектированная система обходится дороже при переделке, чем время, потраченное на проектирование.

---

### Точка 1 — Идея и план *(неделя 1–2)*

**Фаза: Проектирование**

Цель — зафиксировать идею и убедиться, что вы понимаете что делаете и успеете.

**Что приносите:**
- [ ] Описание проекта: что делает, для кого, какую задачу решает (2–3 абзаца)
- [ ] Основные пользовательские сценарии (user stories): *«Как [роль], я хочу [действие], чтобы [ценность]»*
- [ ] Список основных сущностей и связей между ними (можно текстом или таблицей)
- [ ] Список ключевых экранов приложения с кратким описанием что происходит на каждом
- [ ] Задачи разбиты в трекере минимум на крупные блоки с примерными датами

**Какие артефакты стоит оформить на этом этапе:**

| Артефакт | Инструмент | Зачем |
|----------|-----------|-------|
| Описание идеи (Problem Statement) | Notion, Google Docs, README | Фиксирует проблему и целевую аудиторию — основа для всех последующих решений |
| User Stories | Notion, таблица | Помогает думать от пользователя, а не от технологии |
| Mind map идеи и фич | Miro, FigJam | Позволяет увидеть весь продукт целиком и найти связи между частями |
| Список сущностей | Таблица, Notion | Первый черновик будущей ER-диаграммы |
| Список экранов | Notion, таблица | Помогает оценить объём фронтенда и не забыть ни один сценарий |
| Доска задач | Trello, Notion, Linear | Структурирует работу и позволяет контролировать прогресс |

> 💡 Не нужно делать всё идеально — на этом этапе важна ясность, а не полнота. Лучше грубый черновик, чем ничего. Ментор поможет скорректировать.

**Что проверяем:** реалистичность идеи, понимание пользователя и объёма работы, наличие плана

---

### Точка 2 — Проектирование системы *(неделя 2–3)*

**Фаза: Проектирование**

Цель — спроектировать систему на бумаге до написания кода.

**Что приносите:**
- [ ] Wireframes / макеты ключевых экранов (Figma, Excalidraw — любой инструмент)
- [ ] ER-диаграмма с описанием всех сущностей и связей
- [ ] Список эндпоинтов API с описанием входных и выходных данных
- [ ] Структура папок проекта (фронт и бэк)
- [ ] Стартовый фронтенд: роутинг, layout, базовые компоненты

**Что проверяем:** полнота схемы, соответствие идее, нормализация БД

---

### Точка 3 — Бэкенд и база данных *(неделя 4–5)*

**Фаза: Реализация**

Цель — поднять серверную часть, реализовать аутентификацию и подключить БД.

**Что приносите:**
- [ ] Миграции / SQL-схема — база данных создана
- [ ] Реализована аутентификация и авторизация (регистрация, логин, JWT)
- [ ] Базовые эндпоинты работают и протестированы (Postman / Swagger)
- [ ] Конфигурация вынесена в `.env`

**Что проверяем:** безопасность auth, соответствие схемы ER-диаграмме, работа эндпоинтов

---

### Точка 4 — Связка фронта с бэком (MVP) *(неделя 5–6)*

**Фаза: Реализация**

Цель — получить рабочее приложение end-to-end: можно войти и выполнить ключевое действие.

**Что приносите:**
- [ ] Фронт подключён к бэку, данные приходят из БД
- [ ] Реализован основной CRUD для ключевой сущности
- [ ] Работает регистрация, логин, личный кабинет
- [ ] Базовая обработка ошибок на фронте

**Что проверяем:** связность частей, основной пользовательский сценарий работает без ошибок

---

### Точка 5 — Готовность к защите *(неделя 7–8)*

**Фаза: Финализация**

Цель — довести проект до финального состояния и подготовиться к презентации.

**Что приносите:**
- [ ] Полный заявленный функционал реализован
- [ ] Валидация и обработка ошибок везде
- [ ] README с инструкцией запуска
- [ ] Подготовленная презентация / демо проекта
- [ ] (желательно) Тесты, документация API, деплой

**Что проверяем:** готовность к защите, качество кода, соответствие заявленной идее

## Как сдать проект

### 1. Создайте репозиторий

Создайте публичный Git-репозиторий для вашего итогового проекта (GitHub / GitLab — на ваш выбор).

### 2. Оформите README.md в корне репозитория

README — это первое, что видит любой человек, открывший ваш проект. Отнеситесь к нему серьёзно: именно по нему ментор и эксперты будут составлять первое впечатление ещё до запуска приложения.

README должен содержать следующие разделы:

---

#### 📦 Стек технологий

Перечислите все технологии, которые используете: язык, фреймворки, библиотеки, БД, инструменты деплоя и т.д.

Пример:
```
Frontend: React, TypeScript, Zustand, TailwindCSS
Backend:  Node.js, Express, Prisma
Database: PostgreSQL
Other:    Docker, Swagger, Jest
```

---

#### 📋 Описание проекта

Опишите ваш проект как можно подробнее. Чем точнее описание — тем точнее обратная связь от ментора и экспертов. Постарайтесь ответить на следующие вопросы:

- **Идея:** в чём суть проекта, какую задачу он решает и для кого?
- **Функциональность:** что может делать пользователь? Пройдитесь по ключевым экранам и опишите действия на каждом.
- **Уникальные фичи:** что отличает ваш проект? Что вы хотите реализовать сверх базового функционала?

> 💡 Хорошее описание помогает нам скорректировать тему и направление на старте — пока это ещё можно сделать без потерь времени.

---

#### 🗺 План действий

Опишите шаги для достижения финального результата. Структура плана:

1. Крупные вехи с примерными датами
2. Каждая веха декомпозирована на более мелкие задачи

Если пока не знаете, как декомпозировать какой-то шаг — напишите его крупным блоком и вернитесь к декомпозиции позже. Главное — понимать, в какие сроки вы хотите завершить каждый блок.

**Пример структуры плана:**
```
1. Проектирование (до 5 мая)
   - Описание сущностей и связей
   - Wireframes ключевых экранов
   - ER-диаграмма

2. Фронтенд — базовая структура (до 12 мая)
   - Роутинг
   - Layout и базовые компоненты
   - Страницы авторизации

3. Бэкенд и БД (до 26 мая)
   - Схема БД и миграции
   - Аутентификация (JWT)
   - CRUD-эндпоинты

4. Связка фронта с бэком (до 9 июня)
   ...

5. Финализация и подготовка к защите (до 30 июня)
   ...
```

Альтернативно — можете приложить **ссылку на доску задач** (Trello, Notion, Linear) или **диаграмму Ганта** вместо текстового плана.

---

### 3. Пришлите ссылку на репозиторий

После оформления README — пришлите ссылку на репозиторий своему ментору. Это стартовая точка для первой контрольной проверки.

## Критерии оценки

Оценивать будет состав лидов компании и приглашённые эксперты (ВК, Яндекс, банки и другие компании).

Критерии делятся на две группы с разным весом в итоговой оценке.

### ⚙️ Технические критерии *(основной вес)*

Это то, на что эксперты смотрят в первую очередь. Именно здесь формируется большая часть итоговой оценки.

| Критерий | Что смотрим |
|----------|------------|
| Технический стек | Правильное и осмысленное использование React, Node.js, PostgreSQL |
| Качество кода | Читаемость, структура, отсутствие явных антипаттернов |
| Функциональность | Работает ли заявленный функционал end-to-end |
| Тесты | Наличие и качество unit / интеграционных тестов |
| Документация | README, Swagger, схемы — насколько система понятна без объяснений |
| Инженерная работа | Качество схем и артефактов: ER-диаграммы, схемы архитектуры, описание API, wireframes, схемы бизнес-процессов |

### ✨ Критерии впечатления *(дополнительный вес)*

Эти критерии в меньшей степени влияют на итоговую оценку, но именно они создают wow-эффект и запоминающееся впечатление от проекта.

| Критерий | Что смотрим |
|----------|------------|
| Идея и продукт | Есть ли реальная задача, логика продукта, ценность для пользователя |
| UX / UI | Удобство, визуальная аккуратность, продуманность интерфейса |
| Презентация | Структура выступления, умение показать продукт с лучшей стороны |
| Защита | Умение объяснить принятые решения и уверенно ответить на вопросы |

> ⚠️ Часть критериев останется **закрытой** — это сделано намеренно, чтобы вы ориентировались на качество, а не на чеклист.

**Даты защиты:** 4 июля или 11 июля

## Организационные вопросы

### Технические менторы

К каждому студенту будет прикреплён **технический ментор**. Ментор не решает задачи за вас, но помогает с архитектурными решениями, направляет и помогает разблокироваться.

### Продуктовые эксперты

Помимо технического ментора, вы можете обращаться к **продуктовым экспертам** — людям с опытом в продуктовой разработке (PM / PO). Они помогут ответить на вопросы:
- Правильно ли я понимаю свою целевую аудиторию?
- Какие фичи приоритетнее?
- Как сформулировать пользовательские сценарии?
- Есть ли в идее бизнес-смысл?

> Продуктовые эксперты — это не технические консультанты. Приходите к ним с идеями и сценариями, а не с вопросами про код.

### Работа в группах

Можно работать **в паре** (не более 2 человек). При этом:
- Каждый заполняет карточку проекта **отдельно** и указывает партнёра
- Ответственность несут **оба** — если партнёр подведёт, это ваша общая проблема
- Объём ожидаемого функционала при работе в паре выше

### Смена темы

Изменение темы не возбраняется. Однако помните: каждый день после старта, потраченный на смену темы — это день меньше на реализацию. Чем позже меняете тему, тем меньше времени остаётся и тем выше риск не успеть.

## FAQ

**Можно ли использовать Python для бэкенда?**
Да, но только по согласованию и для ограниченного числа проектов. Напишите Евгению Сапову: [@SapovEvgeniy](https://t.me/SapovEvgeniy). Без согласования — Node.js.

**Будет ли помощь по проектам?**
Да. К вам будет прикреплён технический ментор. Также можно обращаться к продуктовым экспертам по вопросам идеи и приоритизации фич.

**Как не потерять время в начале?**
Первые 2–3 дня потратьте на идею, трекер задач и wireframes — это окупится. Не начинайте код до завершения точки 2.

**Что делать, если я не успеваю?**
Сообщайте ментору заранее — не в последний день. Лучше скорректировать объём, чем сдать незаконченное.

**Можно ли сменить тему после выбора?**
Да, но вы сами несёте ответственность за сроки. Поздняя смена темы существенно сокращает время на реализацию.
